# 操作系统——接口

## 什么是操作系统接口

* 连接上层用户和操作系统软件
  * 命令行、图形按钮、应用程序
  * 调用系统调用
* 命令行——一段程序
  * shell也是一段程序
  * 执行程序——创建进程、执行进程
* 图形界面
  * 基于消息机制
    * 鼠标点击，将其放置到消息队列中
    * 编写内核函数`getmessage()`从消息队列中取消息
  * 图形界面是一个包含画图的C程序
* 操作系统接口表示为系统调用
* POSIX IEEE支持的统一接口标准族
  * 提高操作系统的可移植性

## 系统调用的实现

### 回顾一下GDT和IDT表

这些表放置在内存的某个固定位置，由操作系统初始化的时候设置

#### CPU的工作模式

* 实模式
  * CPU上电时，处理器以实模式运行
  * 内存寻址方式和8086相同
    * 段寄存器基地址+偏移
  * 实模式下，所有内存空间均是可读可写可执行的
* 保护模式
  * 段之间的保护机制，防止内存胡乱访问带来的问题
  * 访问的内存空间变大，寻址为4GB

#### GDT——保护模式控制内存访问的方式

* 全局描述表，是一个数组
* 包含以下内容——64bit，寄存器无法存放
  * 段地址
  * 段的最大长度
  * 访问权限
* 寄存器GDTR存放GDT入口地址
* 保护模式下使用基地址+偏移的方式寻址

#### IDT——中断描述表

* 将每个中断和异常编号和一个指向中断处理事件服务程序的描述符联系起来
* 一个大小为8字节的描述符数组
* IDTR存放IDT入口地址
* 保存256个描述符，不超过256个中断
* 运行于保护模式

### 案例——实现一个系统调用`whoami()`打印一个特定的字符串

#### 问题：函数调用和系统调用的区别？

* 函数调用本质上是jmp，但是内核数据不能随意访问，不能随意jmp到内核某个位置
  * 存在安全性和权限问题
* 如何阻止从用户内存jmp到内核内存
  * 硬件设计——区分内核态和用户态，对应的内存是内核段和用户段
  * 内核段寄存器DPL描述目标段的特权级，CPL当前运行程序的特权级，0表示内核级，3表示用户级
  * 操作系统初始化，针对不同的内存段建立GDT表项
  * jmp需要检查是否满足$DPL\geq CPL$，满足才能跳转
* 如何允许用户态进入内核态
  * 中断，int指令将CS中的CPL改成0，进入内核
  * 用户程序进入内核代码唯一方式
  * 系统调用包含
    * 用户程序中包含int指令的代码`int 0x80`
    * 操作系统编写中断处理程序，获得系统调用编号
    * 操作系统根据系统调用编号执行相应代码

#### 代码实例

* 调用`printf("%s",str)`，C语言转化为write所需的缓冲区和字符个数
* 调用库函数`write()`
* 宏展开，调用0x80中断

```c
_syscall3(int,write,int,fd,const char *buf,off_t,count)  // linux/lib/write.c
#define _syscall3(type,name,...) type name(...){__asm__("int 0x80" : "=a" (__res)...} // linux/include/unistd.h
```

实际展开的内容为，3为参数数目

```cpp
int write(int fd,char *,...){
    long __res;
    int 0x80
    mov __NR_write,%eax // 系统调用号
    mov fd %ebx // 系统调用所需的其它参数
}
```

#### 处理 `int 0x80`

* 查IDT表
* IDT表在操作系统启动时初始化，设置处理函数入口点偏移
* `int 0x80`的IDT表中DPL被设置为3，因此用户态才能访问int 0x80中断指令(CPL=3)
* `set_system_gate(n,addr)-->_set_gate(&idt[n],15,3,addr)`
* `_sys_call_table+4*%eax`找到对应的系统屌用函数指针，`%eax`存放的是系统调用号
* 调用I/O驱动，读写设备文件
